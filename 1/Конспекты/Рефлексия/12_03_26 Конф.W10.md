## Разработка через тестирование (TDD)

TDD (Test Driven Development) — методология разработки, при которой тесты пишутся раньше рабочего кода.  
Появилась в рамках XP (Extreme Programming) и позже стала широко использоваться в Agile и Scrum.

Автор познакомился с TDD через Кента Бека, одного из создателей метода. Рабочий цикл при TDD очень короткий — около 30 секунд: постоянно пишется небольшой тест и минимальный код для его прохождения.

# Три закона TDD

1. Нельзя писать рабочий код, пока не написан тест, который падает.
2. Тест пишется только в объеме, достаточном для его падения.
3. Рабочий код пишется только в объеме, необходимом для прохождения теста.

Рабочий цикл:

1. Написать маленький тест
2. Запустить → тест падает
3. Написать минимальный код
4. Запустить → тест проходит
5. Повторить
# Преимущества TDD

## 1. Уверенность
- тесты постоянно запускаются при изменениях
- разработчик уверен, что код работает
- пример: система FitNesse
    - 64 000 строк кода
    - 28 000 строк тестов
    - ~90% покрытия
## 2. Меньше дефектов

Исследования показывают:
- снижение количества багов в 2–10 раз
- подтверждено компаниями:
    - IBM
    - Microsoft
    - Symantec
    - Sabre
## 3. Смелость изменять код

Когда есть надежные тесты:
- разработчик не боится рефакторинга
- код можно улучшать сразу
- система не «гниет» со временем
## 4. Документация

Модульные тесты:
- показывают как использовать код
- являются лучшей низкоуровневой документацией
- содержат реальные примеры использования API
## 5. Улучшение архитектуры

Писать тесты сначала означает:
- код должен быть изолируемым
- уменьшаются сильные зависимости
- архитектура становится более модульной
# Почему тесты нельзя писать потом

Тесты, написанные после кода:
- проверяют только результат
- не влияют на архитектуру
- хуже покрывают поведение системы

Тесты, написанные заранее:
- формируют дизайн
- лучше проверяют систему
# Когда TDD не подходит

TDD не является:
- религией
- универсальным решением
Иногда метод может быть непрактичным, но такие случаи редки.
